# Apex 后端测试方案

## 1. 文档目标

本方案用于定义 Apex 后端在 V1 阶段的测试范围、测试方法、关键用例与通过标准，覆盖 PRD 中本期纳入范围的后端能力：

- 飞书登录换 JWT
- 元数据接口
- 竞争矩阵查询与 tooltip
- 研发泳道查询
- Excel 导出
- 系统健康检查与同步状态

## 3. 测试目标

- 验证应用入口、路由注册、配置加载无语法或导入错误
- 验证 PRD 对应核心接口均已暴露
- 验证飞书鉴权链路从 `code` 到 `user_info` 的服务实现闭环
- 验证导出接口、缓存接口、健康检查与同步状态接口具备可测性
- 验证后端查询性能与并发能力是否满足 PRD 7.1 的目标

## 4. 测试范围

### 4.1 In Scope

- `/api/system/*`
- `/api/meta/*`
- `/api/auth/*`
- `/api/matrix/*`
- `/api/pipeline/*`
- `/api/export/*` 占位接口的可访问性
- 配置与模块导入
- 查询性能、缓存命中、并发压测方案

### 4.2 Out of Scope

- OSS 实际下载链路的长时间稳定性压测
- Redis / MySQL / DuckDB 并发压测
- 导出任务管理完整业务流
- 历史事件视图

## 5. 测试环境

- Python 3.9+ / 3.11 兼容运行环境
- `backend/.venv/` 已安装依赖
- 本地或服务器可访问的 MySQL、Redis、parquet 数据目录

## 6. 测试方法

### 6.1 静态检查

- Python 语法检查
- 应用对象导入与路由枚举

### 6.2 接口冒烟测试

- 健康检查
- 元数据接口
- 鉴权接口参数校验
- 查询接口请求/响应结构校验

### 6.3 联调验证

- 飞书 `app_access_token` 获取
- `code -> access_token -> user_info -> JWT` 链路
- 前端 `/auth/me` 用户信息回读

### 6.4 性能与并发测试

依据 PRD 7.1，后端性能测试需体现以下量化目标：

- 数据查询：复杂多条件筛选 `< 5s`
- 并发能力：`>= 100` 并发用户

建议压测对象：

- `POST /api/matrix/query`
- `POST /api/matrix/tooltip`
- `POST /api/pipeline/query`
- `GET /api/meta/dictionaries`

建议压测场景：

| 编号 | 场景 | 目标 |
| --- | --- | --- |
| PERF-01 | 矩阵复杂筛选冷启动查询 | P95 `< 5s` |
| PERF-02 | 矩阵相同条件重复查询 | 缓存命中后耗时显著下降 |
| PERF-03 | 泳道图查询 | P95 `< 5s` |
| PERF-04 | 50 并发矩阵查询 | 服务稳定，无明显 5xx |
| PERF-05 | 100 并发混合查询 | 核心接口持续可用，错误率可控 |

建议工具：

- `hey` / `wrk` / `ab` 用于接口压测
- 应用日志 + Redis 监控用于辅助确认缓存命中与慢请求

## 8. 通过标准

- Python 语法检查通过
- 应用对象可导入，核心路由全部注册
- 核心接口与 PRD 范围一致
- 飞书联调链路可从前端获取授权码并在后端完成换 JWT
- 性能项需补充量化压测结果后，方可按 PRD 完整验收

## 9. 风险与补测建议

- 当前仓库几乎没有自动化测试文件，测试主要依赖语法检查、接口冒烟和人工联调
- 飞书授权依赖真实应用凭证与发布配置，建议独立维护测试环境配置
- 导出任务管理接口仍为占位，不能按正式功能验收
- PRD 的 `< 5s` 查询和 `>=100` 并发目标无法通过静态检查证明，必须单独执行压测
